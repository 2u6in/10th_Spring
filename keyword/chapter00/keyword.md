- PK, FK란?

  ### 기본키(Primary Key)

  테이블의 행을 식별하는 유일한 값을 가지는 키

    - **유일성**
        - 중복 불가
    - **최소성**
        - 꼭 필요한 최소한의 속성들로만 키를 구성하는 성질
        - 단일 키일 때에는 무조건 만족
    - **불변성**
    - **존재성**
        - NOT NULL

  ### 외래키(Foreign Key)

  다른 테이블의 기본키를 참조하여 두 테이블 간의 관계를 정의하는 역할

- ERD란?

  ### ERD

  Entity Relationship Diagram의 약자로 데이터베이스 구조를 한눈에 알아보고 이후 DB 개발을 더 쉽게 하기 위해 작성한다

  ERD의 핵심은 **Entity, Attribute, Relationship**

    - Entity는 쉽게 말하자면 테이블이라고 말 할 수 있다

  **<설계 순서>**

    1. 와이어 프레임을 보고 필요한 데이터를 생각하며 테이블, 속성 설계하기
    2. PM와 상의하여 글자 수 제한 같은 제약 조건 설정
    3. 정규화를 잘 생각하며 관계 설정하기

- 연관관계란? 그리고 연관관계를 설정하는 방법은?

  ### 연관관계

  데이터베이스에서 테이블과 테이블 사이의 관계를 말한다

    - 관계 종류
        - 1 : 1
        - 1 : N
        - N : M
            - 보통 중간 테이블을 만들어서 관리

    - **연관관계를 설정하는 방법**
        - 테이블 간 **FK를 이용**해서 서로 연관관계를 설정한다

- 정규화란?

  ### 정규화

  데이터 베이스의 중복을 줄이는 것과 이상 현상 방지를 목적으로 테이블을 나누는 과정

    - 정규화 단계
        - **제 1 정규화**

          한 속성에 여러 개의 값이 들어가 있는 경우 별도의 테이블로 분리 → 1 : N 의 관계 만들어짐

        - **제 2 정규화**

          기본키가 2개 이상의 속성으로 구성되어 있을 때 다른 속성들은 기본키의 일부가 아닌 기본키 전체에 의해 특정(종속) 되어야 한다

          ex)

          | 주문 번호 | 상품 코드 | 상품명 | 상품 가격 | 주문 수량 |
                      | --- | --- | --- | --- | --- |
          | 536365 | 22715 | Card Wedding Day | 10.68 | 12 |
          | 536365 | 51914A | Feather Pen Hot Pink | 11.12 | 12 |
          | 536366 | 22633 | Hand Warmer Union Jack | 12.15 | 6 |

          여기서 주문 번호화 상품 코드가 기본 키인 경우

            - (주문 번호, 상품 코드) → (주문 수량) : 제 2 정규화 원칙을 따름
            - (상품 코드) → (상품명, 상품 가격) : 기본키의 일부 속성에 의해 특정(종속) 되므로 제 2 정규화 원칙을 위반함

          → 따라서 상품 코드, 상품명, 상품 가격을 속성으로 가지는 테이블을 생성해서 분리해야함

          (**주문 번호**, 상품 코드, 주문 수량) || (**상품 코드**, 상품명, 상품 가격)

        - **제 3 정규화**

          어떠한 속성이 기본키가 아닌 일반 속성에 종속되어 있다면 이를 별도의 테이블로 분리해야함

          ex) (주문 번호, 주문 날짜, 회원 번호, 이름, 국가)

          → (**주문 번호**, 주문 날짜, 회원 번호) || (**회원 번호**, 이름, 국가)

- 반 정규화란?

  ### 반정규화

  조회 성능 향상과 개발과 운영의 단순화를 위해서 정규화를 통해 나누어진 테이블을 일부 합치거나 중복을 허용하는 기법

  **반정규화 기법**

    - 테이블 병합
        - join이 너무 많아 응답이 느려질 때 관련 테이블을 합치는 방법
    - 테이블 분할
        - 테이블에 데이터나 속성이 너무 많아 무거울 때, 수평(데이터 행) 기준이나 수직(칼럼) 기준으로 쪼개는 방법
    - 테이블 추가
        - 매번 전체 데이터를 가져오기 벅찰 때, 필요한 데이터만 가지고 있는 테이블을 추가로 만드는 방법
    - 계산된 칼럼 추가
        - join이나 연산 시간을 줄이기 위해 다른 테이블의 데이터나 계산 결과를 미리 칼럼으로 추가해 두는 방법

- DB에서의 상속 관계 표현은 어떻게 하는가?

  ### 조인 전략 (Joined Strategy)

  부모 테이블과 자식 테이블을 각각 따로 만들고, 자식 테이블이 부모의 기본키를 외래키이자 기본키로 가지는 방식

    - 정규화 원칙에 잘 맞고 저장 공간도 효율적이지만 join을 많이 사용해야 해서 성능 저하, 쿼리 복잡 등 문제 존재

  ### 단일 테이블 전략 (Single Table Strategy)

  테이블 하나에 부모와 자식의 모든 속성을 합쳐서 만드는 방식

    - 이 때에는 DTYPE 속성을 이용해서 현재 어떤 테이블인지를 표시
    - 테이블마다 안 쓰는 속성은 NULL로 표시

    - join이 필요 없어 조회 속도가 빠르고 쿼리가 단순하지만 NULL이 많이 생겨 낭비되는 공간 존재

  ### 개별 테이블 전략 (Table per Concrete Class)

  부모 테이블을 만들지 않고 각각 자식 테이블에 부모의 속성(공통으로 사용하는 데이터)을 전부 넣는다

    - 부모 타입을 이용해서 전체 데이터를 통합해서 봐야할 때 큰 어려움 존재
        - 보통 거의 사용 X

- 인덱스란?

  ### 인덱스

  추가적인 저장 공간을 활용하여 데이터 베이스 테이블에 저장된 데이터의 검색 속도를 향상 시키는 자료구조

  ex) 10만 개의 식당 메뉴 데이터가 쌓여있을 때 특정 날짜의 특정 식당의 데이터를 빠르게 찾고 싶다면

    ```sql
    CREATE INDEX idx_date_restaurant ON Meals(제공_날짜, 식당_이름);
    ```

  → 만들어진 인덱스

  | 제공_날짜 + 식당_이름 (인덱스) | 실제 데이터 위치(주소) |
      |:---:| --- |
  | 2024-03-01_학생식당 | ID 1번 위치로 가시오 |
  | ... | ... |
  | 2026-03-16_기숙사식당 | ID 99999번 위치로 가시오 |

  이제 이를 이용해서 10만 개의 데이터를 다 뒤지지 않고 인덱스를 이용해 그 위치로 바로 이동해서 데이터를 찾기 때문에 속도가 굉장히 빠르다
