### RESTful API

REST 원칙을 잘 지키며 설계된 API

- REST : 웹 서비스가 어떻게 동작해야 하는지에 대한 설계 방식
    - 자원 (URI), 행위 (HTTP METHOD), 표현 (Representations)으로 구성
- RESTful하다는 건?
    - URI는 행위가 아닌 명사 중심
        - /getUser X → /users O
    - HTTP 메서드로 행위 표현
    - 상태 코드 명확하게 사용
        - 200(성공), 201(생성), 없음(404), 에러
    - 무상태성
        - 서버는 클라이언트 상태를 저장하지 않음
        - 모든 요청은 독립적
        - 매 요청마다 필요한 정보가 다 있어야함

### 멱등성

동일한 요청을 몇 번을 보내든 한 번 요청을 보낸 것과 같은 효과를 가지고, **서버의 상태**도 동일하게 남아있는 성질이다

- 안전이라는 개념과의 차이점
    - 안전 : 몇 번을 요청하든 리소스의 변화가 없다
    - 멱등 : 리소스의 수정이 발생하더라도 몇 번을 요청한 것과 한 번을 요청한 것의 결과가 같다
- get, delete, put은 멱등
    - put은 멱등하지만 매번 다른 상태코드를 return 가능
- post는 멱등적이지 않다
    - POST는 서버로 데이터를 전송하여 새로운 자원을 생성하는 역할
    - 따라서 매번 요청할 때 마다 새로운 자원이 생성된다
- patch는 멱등하지 않지만 put과 동일하게 사용하면 멱등성을 가질 수 있음

**멱등적이지 않은 예시**

- get
    - get 요청을 보낼 때마다 조회수 1 증가 → 멱등x
        - 요청을 보낼 때마다 서버 데이터의 상태는 변한다
        - 조회수 증가 시키는 요청을 PATCH 요청으로 분리해서 해결
- delete
    - 요청을 보낼 때 정확한 id 값이 아닌 last를 삭제하게 함 → 멱등x

### GET

- 서버에 존재하는 특정 데이터를 가져오고 싶을 때 사용
    - 쿼리로 조회할 데이터 선택 가능
- 캐싱 가능
- 멱등성 O

### POST

- 서버에 새로운 데이터를 생성, 추가 할 때 사용
- 주로 요청의 본문(Body)에 전달하고자 하는 데이터를 담아서 전송한다
- 멱등성 X

### PUT

- 서버에 존재하는 데이터를 전부 새로운 데이터로 덮어쓸 때 사용
- 만약 덮어 씌우라고 요청한 데이터가 존재하지 않으면 새로 생성한다
- 구체적인 경로 필요
- 멱등성 O

### PATCH

- 서버에 존재하는 데이터 중 일부만 수정하고 싶을 때 사용
- 멱등성 O/X

### DELETE

- 리소스를 제거하는 메소드
- 멱등성 O
